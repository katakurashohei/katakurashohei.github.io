to-do:表記法と管理

2026-01-07

to-doの管理と実行は、研究活動において重要であるが、意外にも難しい。それは多くの場合良いto-doリストを作るのに失敗するからである。

良いto-doリストを上手く作るための方法を模索してきたが、規格化された表記法と三つの道具（to-do list, done list, to-do app)が重要だということがわかってきた。

to-do list

これがわたしのto-do listの抜粋。これが1つのText Fileにまとまっている。

  • (modular 0.5h) design a fracture test object
  • (modular 1h) sketch UI
  • (modular 2h) assemble a modular half chair
  • …
  • ;;============
  • (modular 1h) make a prototalk
  • (CHI26 0.5h) add evaluation result to extended abstract
  • …
  • ;;============
  • (infill) learn plate theory
  • …

表記法: まず一つ一つのto-doの表記が (project time) verb ... と規格化されている。これで見栄えが良くなり、見てすぐ何のプロジェクトでどれくらい時間が完了させるのに必要かがわかる。timeは一つのto-doの推定作業時間で xh (x hour)で統一してある。to-doはverbで始まる。

now, next, unclear: listは;;============で三つに分かれている。一番上が “now” 今すぐやるべきもの。二番目が “next” 次にやるべきこと（今日でなくても良い）。三番目が “unclear” やらなきゃ行けない気がするが、行動の仕方がわからない、その結果どのくらい時間がかかるかわからないto-doである。

All in one: すべてのto-doは同じファイルの中にあり、一度で見れる状態になっていることである。そうすることによって、例えば"now"が空になりそうで今日は時間があるので"next"からすぐに想定時間内に終わるものを"now"に移動したり、“unclear"がクリアになってきて"next"に移動できたりするからである。これを別ファイルだったり他のアプリに保存しておくと大変面倒になってやらなくなる。ここれらの処理を1日に何度もやるのだから、些細なことのように思えるが、非常に重要なことである。

done list

to-doが完了した時点で、実際かかった時間にxhを書き直す。例えば (modular 1h) sketch UI　は実際には30分=0.5hしかかからなかったとしたら (modular 0.5h) sketch UI とする。そしたらそのto-doをファイルの一番下に移動する。 そうすると例えばこのようになっている

  • 2026_01_01
  • (modular 0.5h) design a fracture test object
  • (modular 0.5h) sketch UI

これで私は今日1.5h仕事をしたということがわかる。もちろんto-doをやるだけが、1日の仕事を思い出しながら、もしくはカレンダーを見ながら1日が生産的だったか振り返る。

ちなみに先ほどの規格化によって（projectで検索することで、そのプロジェクトのdoneリストをみることができ、xh)で検索することで、x時間かかったto-doを見ることができる。後者は特に"unclear” カテゴリーのものの時間を推定するときに有用である。

Things (mac and ios app)

最初はこれをtexifileでやっていたが、to-doのdoneを管理するのは大変面倒であった。そこでアプリを使っている。私はThingsを使っている。checkbox UIは非常に便利で、チェックするとグレーになり、1日が終わると自動でLogBookに移動する。ただThingsの8割の機能は使っておらず、Inbox(to-do list）、LogBook（done list）のみ使っている。

まとめ：実際にto-doを実行するには

まず第一に行動可能なto-doにするということである。できる限り、0.5hにto-doを分解すると言うことが重要であるように思う。行動可能なto-doを作るのはある程度の経験が必要なので、自分なりに試行錯誤していくしかない。

verb（動詞）をto-doの最初に書くのも重要である。verbが思いつくと言うことは行動が可能と言うことであると思う。 しかし動詞がthinkやdoであると行動は曖昧になる。sketch, design, codeなど自分の中で具体的に何をするかわかっている動詞を使うのが良い。動詞もある程度規格化されていたほうが良いだろう。

ps: to-do作り自体が研究のプロセスであるとも言える。何を解くべきかを見極め、実際に解く。 何を解くべきかを把握することが実際に解くよりも難しい、と言うよりも解くことの難しさに影響する。
